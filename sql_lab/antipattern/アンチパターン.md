# アンチパターン

|章・パターン名|目的（やりたいこと）|アンチパターン（間違った手法）|解決策（正しい手法）|
|---|---|---|---|
|第I部：論理設計||||
|ジェイウォーク|1つの属性に複数の値を持たせたい|カンマ区切りの文字列として1つの列に格納する|**交差テーブル（中間テーブル）**を作成し、多対多の関係に正規化する|
|ナイーブツリー|階層構造（木構造）を表現・検索したい|常に「親のID」のみを保持する（隣接リスト）|閉包テーブル、経路列挙、入れ子集合、または再帰クエリ（CTE）を活用する|
|IDリクワイアド|主キーの共通ルールを確立したい|どんなテーブルにも思考停止で「id」という無意味な自動連番列を作る|状況に応じて自然キーや複合キーを活用し、分かりやすい列名にする|
|キーレスエントリ|設計や実装をシンプルにしたい|データベースの「外部キー制約」をあえて設定しない|正しく**外部キー制約（FOREIGN KEY）**を宣言し、データの整合性をDB層で守る|
|EAV|行ごとに異なる可変の属性をサポートしたい|「エンティティ・属性・値」を汎用的な縦持ちのテーブルに詰め込む|サブタイプのモデリング（シングル/具象/クラステーブル継承）やJSON（半構造化データ）を使う|
|ポリモーフィック関連|1つの列から複数の異なる親テーブルを参照したい|1つの外部キー列に「親のID」と「親の種類」の二重の目的を持たせる|交差テーブルを作成するか、共通の基底テーブルを作ってリレーションを単純化する|
|マルチカラムアトリビュート|1つの属性に複数の値を持たせたい（その2）|「tag1,tag2,tag3」のように同じ目的の列を複数定義する|従属テーブル（子テーブル）を作成し、1件ずつの行として縦に保持する|
|メタデータトリブル|データ増大に伴うスケーラビリティを高めたい|「2025年用テーブル」「2026年用テーブル」のようにテーブルや列をコピーして増やす|データベース機能のパーティショニングを使うか、正しく正規化する|
|第II部：物理設計||||
|ラウンディングエラー|小数（実数）を正確に扱いたい|金額などの計算に誤差の出る「FLOAT型（浮動小数点数）」を使用する|誤差のない固定小数点数である**NUMERIC型（DECIMAL型）**を使用する|
|サーティワンフレーバー|列に入る値を特定の種類に限定したい|データベースの列定義（ENUM型やCHECK制約）に選択肢をハードコーディングする|選択肢をデータとして管理する**参照テーブル（マスタテーブル）**を作成し、外部キーで結合する|
|ファントムファイル|画像などの大容量メディアファイルを扱いたい|DBにはファイルパス（文字列）だけを入れ、実ファイルはOSのファイルシステムに置く|状況に応じてBLOB型（バイナリ列）でDB内部に直接格納し、トランザクションの一貫性を保つ|
|インデックスショットガン|クエリのパフォーマンスを最適化したい|闇雲にすべての列にインデックスを張る、あるいは全く張らない|MENTORの原則（測定・解析・テスト）に基づき、効果的なインデックスのみを厳選して管理する|
|第III部：クエリ||||
|フィア・オブ・ジ・アンノウン|欠けている値（不明な値）を区別したい|NULLを一般の値（「0」や空白）として扱ったり、逆に一般の値をNULLの代わりに使う|NULLを「値の不在」を表す一意なものとして正しく扱い、必要な列にはNOT NULL制約を設ける|
|アンビギュアスグループ|グループ内で最大値（または最小値）を持つ行を取得したい|GROUP BY を使うクエリで、集約キーでも集約関数でもない「曖昧な列」をSELECTする|**ウィンドウ関数（ROW_NUMBER等）**や相関サブクエリを使い、結果が一意に決まるクエリを書く|
|ランダムセレクション|テーブルからランダムに1行だけデータを取得したい|ORDER BY RAND() を使ってデータ全体をランダムソートする（大損害の元）|全件ソートを避け、主キーの範囲からランダムなIDを選ぶ、またはオフセットを利用する|
|プアマンズ・サーチエンジン|大量のテキストから全文検索を行いたい|LIKE '%キーワード%' などのパターンマッチ述語を多用する（インデックスが効かない）|DBベンダー独自の全文検索インデックス機能や、専用の外部サーチエンジン（Elasticsearch等）を使う|
|スパゲッティクエリ|SQLの発行回数（クエリ数）を減らしたい|複雑な複数の処理や集計を、超巨大な1つのSQL（ワンステップ）で解決しようとする|分割統治を行う。無理に1つのSQLにまとめず、適切な単位でクエリを分ける|
|インプリシットカラム|SQL文のタイピング数を減らしたい|アプリケーションコード内で SELECT * を使い、列名を暗修（省略）する|"将来の変更に強くするため、必要な**列名をすべて明示的に指定（SELECT col1, col2）**する"|
|第IV部：アプリ開発||||
|リーダブルパスワード|パスワードのリカバリーやリセットを行いたい|パスワードをデータベースに平文（生テキスト）のまま格納する|ソルト（Salt）付きのパスワードハッシュ関数（意図的な高負荷ハッシュ）を使って暗号化して格納する|
|SQLインジェクション|動的なSQLクエリを柔軟に記述したい|ユーザーの入力値をそのまま文字列連結してSQLコードとして実行してしまう|**プリペアドステートメント（バインド変数）**を徹底し、入力値をコードから完全に隔離する|
|シュードキー・ニートフリーク|削除されたデータの「自動連番（ID）の欠番」が気になる|連番の隙間（穴）を埋めるために、わざわざIDの値を詰め直す処理を行う|疑似キーの欠番は埋めないのが鉄則。データの不一致やパフォーマンス低下を招くだけと割り切る|
|シー・ノー・エビル|コードを簡潔に、スマートに見せたい|SQLの実行エラー（戻り値や例外）をチェックせず、正常に動いた前提で処理を進める|戻り値と例外のチェックを厳格に行い、エラーからエレガントに回復・ログ出力する処理を書く|
|ディプロマティック・イミュニティ|アプリ開発のベストプラクティスを採用したい|SQLやデータベースを特別扱いし、バージョン管理や自動テスト、文書化の対象から外す|アプリコードと同様に、SQLやスキーマもバージョン管理（Git）やテストの対象として包括的に管理する|
|さびついた開発標準|ストアドプロシージャのメリットを活かしたい|「すべてのビジネスロジックはストアドプロシージャに書くべき」という古いドグマ（教義）に盲従する|現代のアプリケーションアーキテクチャ（アプリ層とDB層の適切な責務分散）を考慮して柔軟に使い分ける|
|第V部：ボーナス||||
|外部キーの誤った使い方|標準SQLで参照整合性を保ちたい|データ型の不一致、親テーブルにキーがない、複合キーの定義順ミス、SET NULLの不正適用など|各種データベースの特性を理解し、リレーションシップの定義ルールを厳密に守って正確に外部キーを設定する|
