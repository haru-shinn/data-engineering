# まとめ方

### ノートのまとめ方

```
地図
「目的地（やりたいこと）」に対して「どのルート（知識）を使えばいいか」の関連性をまとめる方法。
 例：「テーブルを分割したい（目的地）→ 正規化の理論（ルート）→ 重複削除（アクション）→ 詳細は『〇〇本』P.50参照」と書く。
```

### 具体的な「地図」のまとめ方：3ステップ

#### 1\. ナレッジツリーを「もくじ」にする

ナレッジツリーをノートの最初のページに配置する。

#### 2\. 「参照（リンク）」と「ひとこと」に絞る

本や動画の内容をすべて書くのではなく、「自分にとっての気づき」と「参照元」だけをメモする。

- ノートの例（設計フェーズ）：  
  - トピック： 多対多の関係の解消  
  - 気づき： 中間テーブルを作らないとDBが破綻する。Amazonの「注文」と「商品」の関係がまさにこれ。  
  - 参照： 書籍『〇〇』第2章、YouTube『△△チャンネル』10分〜  
  - 疑問： 3つ以上のテーブルが絡む場合はどうなる？（←後で調べる宿題）

#### 3\. 「失敗の記録」を書き加える

地図において「通行止め」の場所を知ることは非常に重要であるため、実装サイクルで起きたエラーや失敗をメモする。

- メモの例（実装フェーズ）：  
  - 事象： インデックスを貼ったのにクエリが速くならない。  
  - 原因： `WHERE` 句で関数を使っていたため、インデックスが無視された。  
  - 教訓： インデックスは「貼り方」だけでなく「クエリの書き方」とのセットで考える。

---

### 「設計 → 実装 → マネジメント」の連動イメージ

地図を作る際は、各フェーズを切り離さず、「前のフェーズとの繋がり」を矢印で結ぶように意識する。

1. 設計時： 「ここで正規化を崩して（非正規化）スタースキーマにすると、分析は楽になるはずだ。」  
2. 実装時： 「実際にSQLを叩いたら、JOINが減って爆速になった！でも更新（INSERT）は少し遅くなったな。」  
3. マネジメント時： 「このスタースキーマは重要だから、データカタログに『分析用』と明記し、リネージ（由来）を記録しておこう。」

※「なぜその設計にしたのか」という理由（地図のルート選択の根拠）を書き残していく。
